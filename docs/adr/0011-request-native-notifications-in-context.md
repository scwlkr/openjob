# Request native notifications in context

The native mobile client never requests notification permission on first launch and does not use provisional or surprise delivery. After the User's first meaningful Task interaction, OpenJob may show one subtle in-app invitation explaining the value of Task notifications. The operating-system permission prompt appears only after the User explicitly chooses **Enable notifications**; dismissing the invitation or denying permission does not block the app or cause repeated prompts. A permanent Notifications control remains available in Settings. This preserves the existing User-approved Notification Subscription model while adapting its invitation timing to native platform expectations.

Selecting a native Push Notification deep-links to its exact Task inside the Group, resuming that destination after authentication when necessary. If the Task was deleted or the User no longer has access, OpenJob opens the Group when possible and explains that the Task is no longer available. Notification content is private by default: it identifies the event and Group but omits Task text. A per-installation **Show Task text** setting can opt into the preview, subject to stricter operating-system notification privacy settings.

The native app does not set an app-icon badge in its first public release. OpenJob has no unread inbox or other durable attention state, and an open-Task count would not mean unread activity. A badge can be introduced only with a corresponding state model that gives the number one reliable meaning and clears it consistently across installations.
